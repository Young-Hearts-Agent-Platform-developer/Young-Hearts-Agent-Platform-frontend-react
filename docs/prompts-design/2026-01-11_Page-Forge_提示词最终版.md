# Page-Forge 提示词（设计/优化）完整报告
## 报告日期
2026-01-11

## 核心需求
设计一个智能体：接收（1）技术栈文档；（2）页面美术设计文档；并依据输入与 `.instruction` 自动生成可集成的页面代码与交付清单。

## 目标文件上下文详情
- 文件路径： .github/agents/Page-Forge.agent.md
- 文件用途：作为项目内可调用的智能体提示词，用于把设计稿与技术栈映射为可交付的前端页面实现方案。
- 上下文核心：需兼顾工程化产出（文件清单、代码片段、集成步骤）与设计还原（视觉与交互），并严格遵守 `.instruction` 的优先级。
- 适配逻辑：提示词遵循项目技术栈约定（TypeScript/CSS Modules 等），输出文件结构与内容直接可复制到前端项目内。

## 设计/优化详情
### 核心设计思路
- 角色定义依据：以前端工程化交付为目标，选择工程友好的输出格式（files 列表 + manifest）。
- 流程拆解逻辑：先解析约束，再拆组件，接着生成代码与样式，最后给出集成与验收清单，便于快速交付与复测。
- 约束设定原因：明确文件格式和交付模板，避免生成模糊的概念性输出，提高可执行性。

### 优化点（相对于模糊提示词）
|原提示词问题|优化方案（结合上下文）|优化效果|
|---|---|---|
|角色模糊|明确为前端页面生成器智能体，并限定技术栈优先级|生成输出更工程化，易上手|
|输出结构不清晰|定义 `files` 列表与 `manifest.json` 模板|便于自动化写入与集成|

### 验证过程与结果
- 模拟输出示例：基于示例输入，智能体应返回 `files`（含 src/pages/...：组件文件与样式）、`manifest.json`、`integration.md`、`acceptance-checklist.md`。
- 验证维度：
  - 需求匹配度： 符合（明确接收两类文档与 .instruction）
  - 上下文适配度： 贴合（输出遵循项目工程化结构）
  - 角色清晰度： 清晰
  - 流程可执行性： 可执行（解析拆解生成集成说明）
  - 输出合规性： 合规（无外部敏感信息，资源以清单方式呈现）

## 使用指南
- 适用场景：从设计稿到前端实现的自动化初稿生成；为前端开发加速组件落地与集成。
- 调整建议：如需兼容不同样式方案（Tailwind/CSS-in-JS），修改 `.instruction` 指定样式偏好。
- 常见问题：若设计稿缺少交互说明，智能体会生成缺失项清单并请求补充信息。
