# View-Forge 提示词（设计/优化）完整报告

## 报告日期
2026-01-11

## 核心需求
更新提示词以支持两种工作模式：
1. **若没有前端设计方案**：制作全新的方案（保留原有逻辑）。
2. **若存在前端设计方案**：在原来的基础上进行修改（迭代优化模式）。

## 目标文件上下文详情
- **文件路径**：`e:\Project\Web_Development\Young-Hearts-Agent-Platform\Young-Hearts-Agent-Platform-frontend-react\.github\agents\View-Forge.agent.md`
- **文件用途**：定义名为 "View-Forge" 的AI智能体，用于生成前端界面设计文档。
- **上下文核心**：
    - 输入：项目功能文档、前端形式。
    - 输出：Markdown 格式的设计方案（Design System, Sitemap, UI Details）。
    - 角色：资深前端设计师。
- **适配逻辑**：
    - 扩展输入参数，增加“已有设计方案”作为第3个可选参数。
    - 调整执行流程，增加“模式识别”步骤，根据输入决定是走“新建”轨道还是“迭代”轨道。
    - 迭代模式下需保持与原设计风格的一致性。

## 设计/优化详情

### 核心设计思路
1.  **参数扩展**：在 `argument-hint` 中明确 `[已有设计方案(可选)]`，告知用户可以使用此Agent进行后续维护。
2.  **流程分轨**：
    - **从零构建**：保持原有的“分析 -> 设计系统 -> 架构 -> 详细设计”标准流程。
    - **迭代优化**：新增流程，强调“基准对齐”（不随意改风格）和“增量更新”（只改动必要部分）。
3.  **输出一致性**：无论哪种模式，最终输出的都是“完整的设计方案”，而非“差异补丁”，便于开发人员直接使用最新文档。

### 优化点
| 原提示词问题 | 优化方案 | 优化效果 |
| :--- | :--- | :--- |
| 仅支持从零设计 | 引入“从零构建”与“迭代优化”双模式 | 覆盖了项目全生命周期（设计+维护） |
| 输入定义均假定为空白 | 增加可选输入“已有设计方案” | AI 能理解上下文，避免重复设计或风格断裂 |
| 执行流程单一 | 拆分为“轨道A（新建）”与“轨道B（迭代）” | 针对不同场景提供更精准的执行逻辑 |

### 验证过程与结果
- **模拟输出示例（迭代模式）**：
    - **输入**：新功能文档（增加了“夜间模式”）、Web端、旧设计方案。
    - **AI行为**：
        1.  识别到旧方案。

        4.  保持原有页面结构，并在“设置页”增加“主题切换”开关的交互描述。
    - **结果**：输出了一份包含新特性的完整设计文档。
- **验证维度**：
    - 需求匹配度：✓ 符合（支持两种模式）。
    - 上下文适配度：✓ 贴合（沿用 Markdown 输出格式）。
    - 角色清晰度：✓ 清晰（资深设计师，懂维护）。
    - 流程可执行性：✓ 可执行（分轨逻辑明确）。

## 使用指南
- **适用场景**：
    1.  **项目启动阶段**：输入功能文档 -> 产出 v1.0 设计方案。
    2.  **需求变更/迭代阶段**：输入新功能文档 + v1.0 设计方案 -> 产出 v1.1 设计方案。
- **调整建议**：对于微小的文案修改，可直接手动编辑 Markdown；对于涉及业务逻辑变更的界面调整，建议通过此 Agent 进行迭代。
- **常见问题**：
    - **问**：如果我想完全推翻重做怎么办？
    - **答**：不要提供“已有设计方案”参数，或在提示中明确说明“重构”。
