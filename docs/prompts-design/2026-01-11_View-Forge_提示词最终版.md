# View-Forge 提示词（设计/优化）完整报告

## 报告日期
2026-01-11

## 核心需求
将 `View-Forge` 的职能从“生成全局设计方案”转型为“基于全局方案设计特定单页面”。
核心任务是：接收《前端界面设计方案》和《页面名称》，输出该页面的详细视觉与交互设计文档（不含代码）。

## 目标文件上下文详情
- **文件路径**：`e:\Project\Web_Development\Young-Hearts-Agent-Platform\Young-Hearts-Agent-Platform-frontend-react\.github\agents\View-Forge.agent.md`
- **文件用途**：定义“View-Forge”智能体，用于进行细粒度的单页面视觉设计。
- **上下文核心**：
    - 输入变更：不再接收“功能文档”，而是接收已经定稿的“前端界面设计方案”作为规范来源。
    - 输出变更：不再输出单一的大文档，而是输出分散的 `page-designs/*.md` 文件。
    - 边界控制：明确“只负责美术设计，不涉及代码”。

## 设计/优化详情

### 核心设计思路
1.  **角色定位下沉**：从“架构师”变为“执行设计师”。依然保持“资深”水准，但关注点从“Sitemap/Flow”转移到“Padding/Color/Shadow/Interaction”。
2.  **输入依赖性**：设计通过严格解析 `[前端界面设计方案]` 来获取 Design System（色彩、排版、组件规范），确保单页设计与整体风格统一。
3.  **输出结构化**：定义了标准的单页设计文档模板（概述、布局、组件细节、交互），方便开发人员按图索骥。

### 优化点
| 原提示词问题 | 优化方案 | 优化效果 |
| :--- | :--- | :--- |
| output 为全局大文档，难以维护细节 | output 拆分为单页独立文档 | 提高设计颗粒度，便于并行开发 |
| 职责包含架构与细节，过于宽泛 | 聚焦于“视觉与交互细节” | 输出内容更具落地指导意义 |
| 依赖原始需求文档 | 依赖已有的设计方案规范 | 确保了设计的一致性和系统性 |

## 使用指南
- **适用场景**：在《前端界面设计方案》定稿后，开发进入具体页面编码前，使用此 Agent 生成详细的设计说明书。
- **调整建议**：如果总控方案发生色彩/风格变更，需重新运行此 Agent 更新所有子页面设计。
